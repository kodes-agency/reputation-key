# Review 14: Type Safety & Naming Conventions (Re-audit R2)

**Date:** 2026-05-23  
**Scope:** All `src/` — domain, application, infrastructure, server, shared, components, routes.  
**Branch:** `feat/phase-15c-goal-ui`

## Summary

Type safety is generally strong. Branded ID types are consistently used across all contexts with dedicated constructors in `src/shared/domain/ids.ts`. The `any` type is only found in auto-generated files (`routeTree.gen.ts`) and generic hook utilities. `as unknown as` force-casts are isolated to test files and one documented infrastructure repository case. No `@ts-ignore` or `@ts-expect-error` directives found. Non-null assertions (`!`) are mostly in test files. The codebase has good exhaustive `never` checks in domain pattern matching.

## Findings

### 1. [MINOR] `any` type in generic utility hooks (outside tests)

**File:** `src/components/hooks/use-mutation-action.ts` (lines 48, 94)  
**Quote:** `export function useMutationAction<TFn extends (...args: any[]) => Promise<any>>(...)`  
**Rule:** No explicit `any` outside tests.  
**Fix:** Use constrained generics: `(...args: unknown[]) => Promise<unknown>`. The generic `TFn` type parameter already captures the concrete types, so `any` is unnecessary here.

**Also affects:**

- `src/components/hooks/use-action.ts` (lines 30, 48) — `AnyAction = ((...args: any[]) => Promise<any>) & {...}`

### 2. [MINOR] `any` in auto-generated route tree

**File:** `src/routeTree.gen.ts` (42 occurrences)  
**Quote:** `} as any)`  
**Rule:** Auto-generated files are exempt from type safety audits.  
**Fix:** No action needed — this file is generated by TanStack Start's router plugin and should not be manually edited.

### 3. [MINOR] `as unknown as` force-cast in production repository code

**File:** `src/contexts/portal/infrastructure/repositories/portal.repository.ts` (lines 147–152)  
**Quote:** `const rows = result.rows as unknown as ReadonlyArray<{ portal_slug: string; property_slug: string }>`  
**Rule:** No `as unknown as` force-casts outside tests.  
**Fix:** This is documented with an inline comment explaining the Drizzle `execute()` return type mismatch. Consider creating a typed helper or using Drizzle's typed `.select()` API instead of raw SQL, which would eliminate the cast.

### 4. [MINOR] Non-null assertion (`!`) on `jobQueue` in build functions

**File:** `src/contexts/review/build.ts` (lines 60, 77)  
**Quote:** `await input.jobQueue!.add('sync-property-reviews', data, { ... })`  
**Rule:** No non-null assertions on possibly undefined values. `input.jobQueue` is typed `Queue | undefined`.  
**Fix:** Capture in a local const inside the truthy branch: `const q = input.jobQueue; ... q.add(...)` or use a type guard.

**Also affects:**

- `src/contexts/integration/build.ts` (line 111)

### 5. [NIT] Non-null assertions in test files (acceptable but could be improved)

**File:** Multiple test files across `src/contexts/`  
**Quote:** `expect(fakes.goals[0]!.status).toBe('completed')`  
**Rule:** Non-null assertions on possibly undefined. In tests, this is acceptable when the test setup guarantees the value exists.  
**Fix:** Could use `expect(fakes.goals[0]?.status).toBe('completed')` or add explicit length assertions before access. Low priority.

**Affected test files:**

- `src/contexts/goal/infrastructure/event-handlers/on-metric-recorded.test.ts` (8 occurrences)
- `src/contexts/goal/application/use-cases/list-goals.test.ts` (3 occurrences)

### 6. [NIT] Discriminated unions — exhaustive `never` checks are present

**File:** `src/contexts/goal/domain/progress-strategy.ts` (line 125)  
**Quote:** `throw new Error(\`Unhandled goal type: ${\_exhaustive}\`)` 
**Rule:** Discriminated unions should have exhaustive`never`checks.  
**Fix:** Already implemented correctly — the`\_exhaustive`binding ensures TypeScript catches missing cases. The`throw` is a runtime safety net. This is the correct pattern.

**Other locations with correct exhaustive checks:**

- `src/contexts/goal/domain/constructors.ts` (line 140)

## Positive Observations

- **Branded ID types** used consistently — 22 branded types defined in `src/shared/domain/ids.ts` with dedicated constructors. No raw `string` used where branded types expected.
- **Zero `@ts-ignore` or `@ts-expect-error`** found anywhere in the codebase.
- **`as unknown as`** only in test files (mock/fake creation) and one documented repository case.
- **Exhaustive `match().exhaustive()`** from ts-pattern used consistently in server functions and domain logic.
- **`readonly`** applied to all domain types and fields.
- **Functional style** — no `class`, no `this`, no `enum`. Factory functions and records throughout.
- **`Result<T, E>`** from neverthrow used correctly in domain constructors and rules.
- **Generic hooks** (`useAction`, `useMutationAction`) use constrained generics with `TFn` — the `any` is limited to the generic constraint.
