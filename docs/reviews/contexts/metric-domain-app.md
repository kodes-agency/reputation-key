## Findings

### [D15] BLOCKER Use case throws instead of returning Result

```
File:   src/contexts/metric/application/use-cases/record-metric.ts:42
Quote:  throw metricError('unknown_metric_key', `Unknown metric key: ${input.metricKey}`)
Rule:   D15 — No `throw new Error` in domain/application; failures must be returned as Result
Fix:    Return `err(metricError(...))` and change the return type to `Result<MetricReading, MetricError>`.
        The domain's `createMetricReading` already returns `Result`; the use case should follow suit.
        Callers (event handlers, build.ts) must handle both branches.
```

### [D3] MAJOR Use case does not call `createMetricReading` domain constructor

```
File:   src/contexts/metric/application/use-cases/record-metric.ts:41-53
Quote:  if (!BUILT_IN_METRIC_KEYS.has(input.metricKey)) { ... }
        const reading = await deps.metricRepo.insertReading({ ... })
Rule:   D3 — Use cases must "Build domain → Persist". The domain constructor `createMetricReading`
        validates metricKey, value >= 0, and required IDs, returning a Result. The use case
        duplicates the metricKey validation inline and skips the domain constructor entirely,
        meaning value/required-ID validations are never applied.
Fix:    Call `createMetricReading(...)` to validate and build the domain entity, then pass
        the validated reading to the repository. Remove the inline `BUILT_IN_METRIC_KEYS` check.
```

### [D3] MAJOR Use case has no authorization step

```
File:   src/contexts/metric/application/use-cases/record-metric.ts:38-69
Quote:  export const recordMetric =
          (deps: RecordMetricDeps) =>
          async (input: RecordMetricInput): Promise<MetricReading> => {
Rule:   D3 — Use case steps: "Authorize → Load → Check rules → Build domain → Persist → Emit → Return".
        The `RecordMetricInput` carries no role/auth information. CONTEXT.md states `metrics:write`
        is "system-only (internal event handlers)", but the use case itself performs no permission
        check at all. Even internal-only use cases should accept an auth context parameter to
        enforce system-only invocation, or at minimum document the intentional omission.
Fix:    Add an `authContext` field to `RecordMetricInput` (or `RecordMetricDeps`) and verify the
        caller has `metrics:write` or is a system principal. Alternatively, if this is deliberately
        system-only, add a code comment and consider renaming to `recordMetricInternal`.
```

### [D3] MAJOR Use case does not generate its own reading ID

```
File:   src/contexts/metric/application/use-cases/record-metric.ts:45-53
Quote:  const reading = await deps.metricRepo.insertReading({
          organizationId: input.organizationId, ...
Rule:   D3 / D11 — "Domain-generated IDs". The use case delegates ID generation to the repository
        (the repo's `insertReading` returns `MetricReading` with `id`). Per architecture, the
        domain layer should generate IDs (via an `IdGenerator` port or in the constructor) and
        the repository should receive a fully-formed entity. The `createMetricReading` constructor
        already requires an `id` input, but the use case never calls it.
Fix:    Add an `idGenerator` port to `RecordMetricDeps`. Generate the ID in the use case and pass
        it to `createMetricReading`. The repo should receive the complete domain entity.
```

### [D2] MAJOR Event constructor uses `crypto.randomUUID()` directly instead of IdGenerator port

```
File:   src/contexts/metric/domain/events.ts:33
Quote:  eventId: crypto.randomUUID(),
Rule:   D11 — Domain must not import Node crypto directly; use an IdGenerator/Clock port for
        deterministic testability. While `occurredAt` is passed in (good), `eventId` is generated
        via the global `crypto.randomUUID()` inside the domain layer, making tests non-deterministic.
Fix:    Accept `eventId` as a parameter (generated by the use case via an IdGenerator port), or
        accept an `idGenerator` function in the constructor. This aligns with the Clock pattern
        already used for `occurredAt`.
```

### [D11] MINOR Domain constructors file imports `ok`/`err` from neverthrow — not a pure domain concern

```
File:   src/contexts/metric/domain/constructors.ts:13
Quote:  import { ok, err, type Result } from 'neverthrow'
Rule:   D11 — Domain purity. neverthrow is a library, not a framework or I/O dependency, so this
        is borderline acceptable. However, the events file and errors file avoid it. If the project
        convention is to keep domain completely free of external packages, this should use a
        simple `{ ok: true, value } | { ok: false, error }` discriminated union instead.
Fix:    Acceptable as-is if the project uses neverthrow pervasively in domain. Flag for team
        consistency check — if other contexts avoid it in domain, this should too.
```
