# BQC-4.6 — No-Fallback Fault Proof: US Cell Outage Report

**Date:** 2026-07-18 · **Slice:** BQC-4.6 · **Phase:** `phase-bqc4-regional-execution.md` §4.6, §6 ("Target outage never invokes a fallback adapter")
**Proof suite:** `src/shared/routing/infrastructure/repositories/no-fallback-faults.test.ts` (integration project: real PostgreSQL + real Redis/BullMQ — no mocks for the machinery under proof)
**Result:** 6/6 tests green against the **unmodified** 3.x/4.1–4.5 machinery. **Zero defects found; zero machinery changes made.**

## 1. Method

Faults were injected into the US target cell (the only approved cell — ADR 0048) at four seams, with the system observed end-to-end through the real production wiring:

- **Queue unavailable at publish** — the outbox relay ran against a BullMQ `Queue` whose connection pointed at a dead TCP port (`lazyConnect` + offline queue disabled + no retry, so `queue.add` rejects fast). The shared Redis was never killed or restarted; the healthy handle used the **same queue name** (the cell's queue — no second queue or region exists).
- **Database unavailable at dispatch** — the routing adapter's `loadPropertyRouting` port threw `ECONNREFUSED` (the drizzle select failing against a down PostgreSQL) inside the 4.2 routing gate, first via the dispatch closure directly, then through a **real BullMQ worker** running `createGatedJobHandler` — the same wiring `src/worker/index.ts` performs.
- **Provider (GBP) down** — the publish-reply handler ran against a `googleReviewApi` port rejecting with the real error classes the integration adapter produces: `gbp_api_error` with `status: 503` (twice), then a request timeout (`AbortError`) on the final attempt — a provider that first serves server errors, then stops responding.
- **Denied Europe property** — a real property row with `processing_region='europe'` read through the **real drizzle routing adapter** (`createPropertyRoutingLoader`) and the real `ProcessingRouter`.

Determinism: relay polls and dispatch closures were invoked directly (never interval-driven); lease expiry was simulated by backdating `lease_expires_at` (no wall-clock sleeps); worker completion was awaited via bounded condition polling (`vi.waitFor`); queue names are unique to the suite and obliterated before/after; all org-scoped rows cleaned up.

## 3. The no-fallback statement

At **no point** in any injected condition did a failure invoke a fallback adapter, fallback queue, fallback endpoint, or fallback region:

- (a) retried on the **same** queue name; no second queue was constructed as a reroute (the dead and healthy handles are one logical queue — availability changed, topology did not).
- (b)/(e) the job failed, retried, parked, and resumed on its **own** cell queue (`originalQueue` = the enqueueing queue in every envelope).
- (c) all three provider calls went to the **one** provider binding the cell has; there is no alternate-provider code path — the 4.3 architecture guards make this structural, not behavioral: `providerRefForCell` has no provider for a non-approved cell (`src/shared/architecture/provider-target-selection.test.ts`), and direct/global queue or provider construction is contracted away (`adr-0048-presence.test.ts`, `entry-point-catalogue` / `event-job-catalogue` guards).
- (f) a denied-region job is quarantined by the 4.2 gate (`routing_blocked:region_denied`) — the wrong-cell/blocked quarantine is the enforcement point (`src/shared/jobs/delayed-execution-gate.ts` `enforceJobRouting`); redrive without approval blocks again.

**Rollback posture (§7 of the phase doc) is what (a)+(b)+(e) demonstrate:** pause the workload in its approved cell and preserve the jobs — never route elsewhere.

## 4. Gaps found and fixes applied

**None.** The one open question going into this slice — whether the 4.2 routing gate propagates or swallows a database failure from the routing adapter's read — was resolved by code inspection and then proven by test (b): `enforceJobRouting` and `createProcessingRouter.resolve` contain no error handling around `loadPropertyRouting`, so the failure propagates out of the dispatch closure as a BullMQ-visible job failure (transient semantics: retry, then 3.6 dead-letter quarantine at exhaustion). This is the correct behavior — a DB outage is transient infrastructure failure, not a policy revocation — and matches the 3.2 gate's `deny_retry` philosophy for unavailable policy reads. No code changed in this slice.
