# Beta feedback widget options for RepKey

**Research date:** 2026-07-15
**Scope:** authenticated beta testers reporting bugs and suggestions from the RepKey web app
**Sources:** first-party product documentation, pricing/security pages, official SDK/source repositories, and web specifications only

## Executive decision

Build a small RepKey-owned **Beta feedback** launcher, but use **Sentry User Feedback + Session Replay** as its capture and diagnostic backend. This is the best fit because RepKey already has `SENTRY_DSN` and trace-sampling configuration reserved for the planned observability phase, uses request IDs and Pino tracing, and now has an official `@sentry/tanstackstart-react` SDK (beta as of the JavaScript SDK 10.55.0 release). Sentry's API lets RepKey keep its own React form while submitting feedback with an associated replay; its widget also supports a cropped screenshot. The result is one debugging spine for tester reports, frontend errors, server traces, releases, and source maps rather than a second isolated bug database.

Do **not** build screenshot capture, annotation, network interception, or session replay from scratch for the beta. The browser screen-capture standard requires an active user action, a permission prompt, and user selection of a surface on every capture; capture can expose cross-origin confidential content. Building a safe annotator, recorder, payload scrubber, retention workflow, and developer viewer is a product in itself ([W3C Screen Capture specification](https://www.w3.org/TR/screen-capture/)).

If annotated video is a must-have before Sentry is wired, run a two-week **Jam Recording Links** pilot behind the same launcher. Jam is the strongest tactical option: inexpensive, no tester account or extension is required, and a custom-domain recording can include console/network logs and automatic blurring. It is not the preferred long-term system because Recording Links are link-accessible rather than membership-restricted, Jam stores recordings in the central US and cannot redirect them to RepKey-owned storage, and it is capture-first rather than a good text-only suggestion inbox.

## RepKey constraints and opportunities

- The app is React 19 + TanStack Start SSR with file-based authenticated routes. The latest official Sentry JavaScript release lists `@sentry/tanstackstart-react` as beta and recommends client-entry initialization, so framework fit is now direct ([official Sentry JavaScript 10.55.0 release](https://github.com/getsentry/sentry-javascript/releases/tag/10.55.0)).
- The repository already declares `SENTRY_DSN` and `SENTRY_TRACES_SAMPLE_RATE`, and its roadmap explicitly defers full Sentry wiring to the observability phase. Adding a feedback form on top of Sentry advances an existing direction rather than introducing a separate platform.
- RepKey already has request-scoped IDs, Pino spans, Better Auth identity/organization context, beta allowlist controls, PostgreSQL/Drizzle, and an S3 storage adapter. Those are useful for a custom launcher and correlation, but they do not provide browser replay, redaction, or a developer replay viewer.
- This is a multi-tenant hospitality product. Screenshots, DOM replay, console output, and request payloads can contain guest names, emails, review text, OAuth tokens, and organization data. The beta must use pseudonymous IDs, pre-send masking, payload allowlists, explicit consent, and short retention.
- The Git remote is GitHub and repository standards direct unresolved work to GitHub Issues. GitHub is therefore the default downstream tracker; its official REST API also supports server-side issue creation if RepKey later needs a custom triage automation ([GitHub Issues REST API](https://docs.github.com/en/rest/issues/issues#create-an-issue)).

## Option-by-option findings

### 1. RepKey-owned custom widget

**What it can do well.** A native launcher can match the existing design, appear only for beta tenants, prefill the signed-in tester, distinguish **Bug / Suggestion**, enforce structured fields, attach RepKey-specific context, keep data in the existing EU-controlled application stores, and route accepted reports to GitHub. Tester accounts do not need access to the engineering tracker.

**What must be built.** The practical custom scope is much larger than the form: capture and crop; annotation; S3 upload lifecycle; redaction; console and `fetch`/XHR instrumentation; replay buffering; browser compatibility; payload limits; report viewer; duplicate detection; retention/deletion; alerting; and tracker synchronization. Browser-native display capture always requires fresh user selection and cannot silently capture the current page ([W3C](https://www.w3.org/TR/screen-capture/)). A DOM screenshot library avoids the browser prompt but has predictable cross-origin, iframe, canvas, font, and CSS fidelity gaps.

**Verdict.** Build the **intake UX**, not the capture infrastructure. A fully custom solution is justified only if vendor data processing is unacceptable or feedback becomes a core RepKey product capability.

### 2. Sentry User Feedback + Session Replay

**Tester experience.** Sentry can inject its own button/dialog or attach the dialog to a RepKey button. More importantly, RepKey can render a completely custom form and call `Sentry.sendFeedback(..., { includeReplay: true })`; user name/email can be prefilled from the Sentry user context ([User Feedback configuration](https://docs.sentry.io/platforms/javascript/guides/svelte/user-feedback/configuration__v7.x)). The screenshot control is enabled by default in modern JavaScript SDKs and includes cropping, but not freehand/arrows/text annotation ([Sentry screenshot announcement](https://sentry.io/changelog/user-feedback-widget-screenshots/)).

**Developer context.** Feedback can carry a replay, while Sentry issues provide stack traces, breadcrumbs, tags, attachments, console/network/DOM events, and related traces ([Issue Details](https://docs.sentry.io/product/issues/issue-details/)). RepKey can attach release, route, beta cohort, hashed organization/user ID, role, and a correlation ID as tags/context; Sentry's TanStack Start guide documents user, tags, contexts, breadcrumbs, attachments, and scoped enrichment ([event enrichment](https://docs.sentry.io/platforms/javascript/guides/tanstackstart-react/enriching-events)). This is the only option here that naturally unifies voluntary feedback with automatic error/performance monitoring across client and server.

**Integrations.** Team and above include third-party integrations. Sentry supports GitHub issue creation/linking and source/commit context, Jira issue linking/sync, Slack alerts, and a Linear-built cloud integration that creates/links Linear issues and resolves/reassigns them bidirectionally ([Sentry integration provider API](https://docs.sentry.io/api/integrations/get-integration-provider-information/), [Linear's Sentry integration docs](https://linear.app/docs/sentry)). The Linear integration does not support self-hosted Sentry.

**Privacy and hosting.** Replay must be configured conservatively; Sentry provides client- and server-side scrubbing controls, IP scrubbing, and granular replay access ([organization privacy settings API](https://docs.sentry.io/api/organizations/update-an-organization/)). Every plan can select Sentry's Germany data-storage location ([EU region announcement](https://sentry.io/changelog/data-storage-location-in-germany-is-generally-available/)). Sentry also maintains a self-hosted distribution with most Business-plan capabilities, but current releases use the non-OSI FSL-1.1-Apache-2.0 license and come with no guarantees or dedicated support plus significant operational requirements ([self-hosted Sentry](https://develop.sentry.dev/self-hosted/)).

**Price as of 2026-07-15.** Developer is free but limited to one user. Team is **$26/month billed annually**, with unlimited users, third-party integrations, 50 replays/month, 1 GB attachments, and up to 90-day lookback; Business is **$80/month billed annually**. Usage above included quotas is pay-as-you-go ([Sentry pricing](https://sentry.io/pricing/)). User-feedback screenshots consume the attachment quota.

**Verdict.** Best strategic fit and best debugging value per integration added. Main gaps are screenshot annotation and a less product-management-oriented feedback inbox.

### 3. Jam.dev

**Tester experience.** Team members can use the extension for screenshots, annotated recordings, and a two-minute DOM Instant Replay. External/internal testers can instead use a reusable Recording Link with no Jam account, extension, or download; a same-domain/custom-domain setup can be launched from an in-product button. With Jam's scripts installed on that domain, the recording includes console logs, network requests, click/key events, device metadata, and browser-side automatic blurring ([Recording Links](https://jam.dev/docs/recording-links), [Instant Replay](https://jam.dev/docs/instant-replay), [DevTools](https://jam.dev/docs/devtools)).

**Developer workflow.** Jam has direct GitHub, Jira, Linear, Slack and other issue-tracker integrations. A created issue carries or links to the capture and its technical context; Linear/Jira fields can be filled without leaving the capture flow ([Linear integration](https://jam.dev/docs/linear), [Jira integration](https://jam.dev/docs/jira)).

**Privacy and hosting.** Recording requires explicit start and submit actions. Custom-domain auto-blur happens in the browser, but Jam documents brief unblurred flashes as a possible limitation during page load/reflow. Recording Links are accessible to anyone who has the URL and cannot be restricted to particular users. Jam is SOC 2 Type II/GDPR compliant, encrypts data at rest/in transit, stores it in GCP Central US, does not support customer-owned recording storage, and reserves custom retention/access controls for Enterprise ([Jam security](https://jam.dev/docs/security)).

**Price as of 2026-07-15.** Free: 30 Jams/month, five recording links, five-minute recordings, five creator seats, unlimited viewers. Team: **$14 per creator/month billed annually**, unlimited Jams, 150 recording links/month, 15-minute recordings, 50 creator seats, unlimited viewers. Enterprise is custom ([Jam pricing](https://jam.dev/pricing/)).

**Verdict.** Best low-cost manual bug-capture pilot and excellent for an internal QA team. Weaker for text-only ideas, closed tester authorization, EU data residency, and long-term correlation with backend traces unless paired with Sentry.

### 4. Usersnap

**Tester experience.** Purpose-built in-app widgets support annotated screenshots, screen + voice recording, flexible forms, surveys, and follow-up conversations. URL, location, browser, and screen metadata are automatic. Professional and higher add automatic client-side JavaScript console and network-error capture; this is useful but is not the same as a full passive DOM session replay ([Usersnap pricing/feature comparison](https://usersnap.com/pricing)).

**Developer workflow.** Native integrations cover GitHub, Jira Cloud/Server, Linear (including two-way status sync), and Slack, plus webhooks/Zapier ([integration list](https://help.usersnap.com/docs/3rd-party-integrations-customer-feedback-projects), [Linear sync](https://help.usersnap.com/docs/linear)). This is the richest option for mixing bug reports, suggestions, surveys, a feedback board, and tester follow-up.

Usersnap's current console-recorder documentation says captured logs cannot presently be exported or passed through integrations, so developers may need to open Usersnap to inspect them. Verify this behavior during a trial rather than assuming the tracker ticket is self-contained ([console recorder](https://help.usersnap.com/docs/console-recorder-feature)).

**Privacy and hosting.** Usersnap is cloud-only, stores customer data in AWS Frankfurt/EU, encrypts it in transit and at rest, is GDPR-oriented, and offers pre-capture PII blackouts. Its security page says screenshots may contain names, email, browser/OS/screen/URL and sometimes IP/location data. The widget uses local storage but no cookies ([Usersnap security](https://usersnap.com/security), [widget security details](https://help.usersnap.com/docs/security)).

**Price as of 2026-07-15.** A free entry allows the first 20 feedback items. USD annual monthly-equivalents are **Starter $49**, **Growth $109**, **Professional $189**, and **Premium from $369**; month-to-month prices are $59/$129/$229/from $449. Because console/network-error capture starts at Professional, the developer-grade configuration starts at **$189/month billed annually** ([Usersnap pricing](https://usersnap.com/pricing)).

**Verdict.** Best product-feedback/tester communication suite, but expensive and duplicative if RepKey mainly needs beta bug diagnosis.

### 5. Marker.io

**Tester experience.** Reporters need no account. The embedded widget provides high-fidelity screenshots and annotations, customizable forms, follow-up, and a browser extension alternative. Team adds session replay, console logs, network requests, and custom metadata; browser/OS/page/viewport are captured automatically ([Marker.io product tour](https://marker.io/features)).

**Developer workflow.** Direct GitHub, Jira, Linear, and Slack integrations are available; Jira/Linear support issue/status/comment synchronization. The initial screenshot, description, form fields, reporter and environment details reach the tracker, while full console/network/replay data stays in Marker.io behind links, with summaries in the ticket ([integration behavior](https://help.marker.io/en/articles/14430683-why-are-console-logs-network-requests-and-replays-linked-instead-of-attached), [integrations](https://help.marker.io/en/collections/106758-integrations)).

**Privacy and hosting.** Marker.io is cloud-only in AWS Ireland/EU, GDPR compliant and SOC 2 Type II audited, with TLS 1.2+ and AES-256. Optional replay buffers are deleted within about 15 minutes if no report is submitted; submitted data is kept until the customer deletes it and backups age out within 90 days. Client-side network filtering and capture masking exist, but pricing places Sensitive Data Masking on the custom Business tier ([privacy/security overview](https://help.marker.io/en/articles/12383990-data-privacy-security-overview)).

**Price as of 2026-07-15.** Starter: **$39/month annually or $59 monthly**, three users, screenshots/annotations and basic metadata. Team: **$149/month annually or $199 monthly**, 15 users, session replay, console/network developer tools and custom metadata. Business is custom. Reporters and feedback volume are unlimited, subject to page-view/project limits ([Marker.io pricing](https://marker.io/pricing)).

**Verdict.** Best polished website bug-reporting workflow if annotated capture is non-negotiable. Cost and Business-tier masking make it harder to justify for an early beta.

### 6. Gleap

**Tester experience.** A broad in-app support/feedback SDK combines structured bug reporting, annotated screenshot, console/network logs, 30–60 second replay, device metadata, chat, surveys, roadmap, changelog, and feature requests. Password/input fields and configured selectors can be masked in the browser ([in-app bug reporting](https://www.gleap.io/product/in-app-bug-reporting)). It supports web plus native iOS/Android, which matters only if RepKey adds native clients.

**Developer workflow.** Native Jira, Slack, Linear, and GitHub integrations are available, with webhooks and many additional connectors ([Gleap integrations](https://www.gleap.io/integration-categories/native)). Integrations require Team or above. Gleap is much broader than a beta widget and can replace support chat, product-feedback board, surveys, roadmap and release notes.

**Privacy and hosting.** Gleap is cloud-hosted in Frankfurt/EU by default, GDPR compliant and SOC 2 Type II audited; US residency and enterprise contractual/security features are on Enterprise. Its trust center provides a DPA and subprocessor list. No self-hosted offering is documented ([Gleap trust center](https://www.gleap.io/trust)).

**Price as of 2026-07-15.** Annual monthly-equivalents: Starter **$49** (one seat/project, bug reporting), Team **$149** (unlimited seats/projects and third-party integrations), Pro **$299**, Enterprise from **$999**. Month-to-month is $59/$179/$359/from $1,199. AI usage is additional token-based billing ([Gleap pricing](https://www.gleap.io/pricing)).

**Verdict.** Strongest all-in-one customer-support/product-feedback option, but it buys far more platform than this beta needs. Choose it only if RepKey also intends to consolidate chat, surveys, roadmap, changelog, and support.

### RepKey recommendation if open source is a requirement

1. **Best diagnostic route:** use **OpenReplay** for passive replay and developer context, keep the proposed small RepKey-owned Bug/Suggestion form, and send accepted bugs to GitHub after triage. This is the strongest genuinely open-source substitute for Sentry, but it adds meaningful infrastructure and still needs a separate intake form.
2. **Best lightweight visual route:** pilot **BugDrop** only if the team is comfortable with its youth and GitHub-as-database model. Self-host the Worker, require a short-lived RepKey auth token, keep screenshots optional, mark sensitive DOM regions, route to a private triage repository, and prevent the screenshot branch from triggering CI. It does not replace error monitoring or replay.
3. **Best split workflow:** use **GlitchTip** for low-cost error/traces/logs plus **Formbricks**, **Fider** or **ClearFlask** for suggestions. This is open source and operationally understandable, but two systems and the missing replay make debugging slower.
4. Do not self-host Sentry merely to save the hosted Team fee: it is fair-source rather than OSI open source, and its infrastructure cost is far higher than the subscription. Do not begin a new Highlight.io deployment after its hosted-service shutdown.

For the current beta, hosted Sentry Free remains the lowest-risk default. If vendor hosting is disallowed, start with **OpenReplay + a RepKey-owned form**; if the beta is small and most bugs are visual, **self-hosted BugDrop + GlitchTip** is the cheaper, more modular experiment. Keep suggestions out of automatic GitHub issue creation in either design.

## Recommended MVP architecture

### Tester flow

1. Mount a small **Beta feedback** button in the authenticated app shell only when the server-confirmed organization is in the beta cohort. Keep a global kill switch.
2. Open a native RepKey dialog with **Bug** and **Suggestion** paths.
3. For a bug, collect: short title, expected result, actual result, optional reproduction steps, impact, and consent toggles for screenshot/replay. Do not require technical fields.
4. For a suggestion, collect: desired outcome, current friction, and importance. Screenshot/replay remains optional and off by default.
5. Prefill the known tester identity but display exactly what will be submitted. Confirm success with a stable report ID and a human response expectation.

### Capture and correlation

- Initialize the official TanStack Start Sentry client only for allowed beta/production environments. Use Sentry's Feedback and Replay integrations with conservative sampling; attach replay on explicit feedback even if continuous sampling is low.
- Before submission, add scoped, allowlisted metadata: `feedback_type`, app release/commit, route template (not raw sensitive query strings), browser/viewport, locale/time zone, tester role, beta cohort, pseudonymous user ID, pseudonymous organization ID, and a generated feedback correlation ID.
- Propagate that correlation ID in the submission/server call and Pino logs so developers can search backend logs without putting log bodies into the browser report. Never send Better Auth cookies, OAuth tokens, raw request/response bodies, guest review text, or full application state.
- Default Sentry Replay to mask text and inputs; explicitly unblock only audited, non-sensitive selectors. Disable request/response body capture unless an endpoint and field allowlist is approved. Use the Germany region and a short beta retention target.

### Triage and developer loop

1. Feedback lands in a separate Sentry project/environment tagged `beta-feedback`.
2. A GitHub integration creates issues only after triage, or an alert rule creates them automatically for high-impact bugs. Suggestions should be batched/deduplicated before issue creation so GitHub does not become an unfiltered inbox.
3. GitHub issue template includes the Sentry feedback/replay link, report ID, release, route, expected/actual behavior, and privacy-safe metadata. Slack notification is for new high-impact reports only.
4. Close the loop manually during beta: acknowledge, request clarification through the agreed tester channel, and notify the tester when fixed. Track `new → triaged → accepted/duplicate/declined → fixed` in Sentry/GitHub rather than adding another RepKey admin screen in MVP.

### Safety gates before launch

- Verify screenshot/replay masking on login, security, integrations, organization/member, review, inbox, and guest-feedback pages with realistic sensitive test data.
- Exclude auth, OAuth callback, password/reset, token, and file-upload payloads from network capture.
- Hash or replace tenant/user identifiers; never tag raw organization names or emails unless the beta privacy notice explicitly permits them.
- Add abuse/rate limits and submission size/type limits even though the widget is authenticated.
- Document consent, subprocessors, storage region, retention, deletion, and who can view replays. Restrict replay access to the smallest developer group.
- Exercise deletion/export and the global widget/replay kill switch before admitting testers.

## Rollout recommendation

- **Week 1:** wire core Sentry error monitoring, releases/source maps, privacy rules, and EU project; validate on staging.
- **Week 2:** ship the custom beta launcher using `sendFeedback` with screenshot/replay consent; connect GitHub and a low-noise Slack alert.
- **Weeks 3–4:** measure report completion, reports needing clarification, reproduction success, duplicate rate, time-to-triage, and replay usefulness.
- **Decision point:** if testers repeatedly need arrows/video narration, pilot Jam for those reports. If the team also needs surveys, feedback voting, roadmap/changelog, and customer support, evaluate Gleap or Usersnap as a broader platform. If neither occurs, keep the Sentry-backed launcher and avoid a second feedback system.
