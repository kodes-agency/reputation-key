# Review: Phase 14.5 Test Coverage & Edge Cases

## Summary

The Phase 14.5 test suite covers the core staff attribution flow reasonably well — referral code generation, resolution, scan recording with referral codes, and metric event handler pass-through of `staffId`. However, the review uncovered **1 critical bug** (staffId is NOT actually passed through to ratings/feedback despite the server function calling `getStaffIdForSession`), **3 high-severity gaps** (missing empty-string handling, no integration test, weak value assertions on staffId), and **several medium/low issues** around edge cases and untested code paths.

**Issues by severity:** P0: 1 · P1: 3 · P2: 5 · P3: 4

## Critical Issues (P0)

### P0-1: `submitRating` use case ignores `staffId` — hardcodes `null`

**Files:** `src/contexts/guest/application/use-cases/submit-rating.ts:48,66`

The server function `public.ts:153` correctly calls `getStaffIdForSession(sessionId)` and passes the resolved `staffId` to `submitRating`. However, `SubmitRatingInput` does **not** include a `staffId` field — the type has no `staffId` property at all. Inside `submitRating`, `buildRating` is called with `staffId: null` (line 48) and the emitted `ratingSubmitted` event also hardcodes `staffId: null` (line 66).

This means **no rating or feedback will ever carry a staffId**, completely defeating the staff attribution feature for the rating/feedback pipeline. The scan pipeline works; the rating/feedback pipeline is broken.

**Impact:** All ratings and feedback submitted through the portal will have `staffId: null`, making staff performance attribution impossible for these interactions.

**Same issue in `submitFeedback`:** `src/contexts/guest/application/use-cases/submit-feedback.ts:46,64` — `staffId` is hardcoded to `null` and `SubmitFeedbackInput` has no `staffId` field.

## High Issues (P1)

### P1-1: `recordScanWithRef` treats empty string `""` as truthy — will query repo with empty code

**File:** `src/contexts/guest/application/use-cases/record-scan-with-ref.ts:26`

```ts
if (input.referralCode) {
```

When `referralCode` is `""` (empty string), this condition is falsy, so it correctly skips the lookup. However, the Zod schema in `public.ts:22` defines `referralCode: z.string().nullable().optional()` — a valid input is `referralCode: ""` which will be passed as `""`. While the truthiness check happens to work, this is fragile and should be explicitly `if (input.referralCode !== null && input.referralCode !== '')` or the schema should strip empty strings. **No test covers the empty string case.**

### P1-2: No integration test for the full staff attribution flow

There is no end-to-end or cross-context test that exercises: generate referral code → build assignment → resolve code → record scan with referral code → retrieve staffId for session → verify staffId propagated to metric. Each piece is tested in isolation but the critical integration between contexts (staff → guest → metric) is unverified.

### P1-3: Event handler tests only assert `staffId: null` — the "with staffId" path is never tested

**Files:** All metric event handler tests (`on-scan-recorded.test.ts`, `on-rating-submitted.test.ts`, `on-feedback-submitted.test.ts`)

Every event handler test constructs events with `staffId: null` and asserts `staffId: null` in the captured readings. There is **no test** that verifies the correct `staffId` value is passed through when it's non-null. For example, in `on-scan-recorded.test.ts:48`, the event always has `staffId: null`, and the assertion at line 59 checks `staffId: null`. A bug that always sets `staffId: null` regardless of input would pass all existing tests.

**Recommended:** Add at least one test per handler where `staffId: staffId('staff-1')` and assert the exact value.

## Medium Issues (P2)

### P2-1: `scanEventFromRow` uses unsafe `as` cast for `source` field

**File:** `src/contexts/guest/infrastructure/mappers/guest.mapper.ts:19`

```ts
source: row.source as ScanEvent['source'],
```

If the database contains a source value not in `'qr' | 'nfc' | 'direct'` (e.g., due to a migration or manual insert), this will silently produce an invalid domain object. No validation or test covers this edge case. The mapper test only tests `scanEventToRow` (domain → row), not `scanEventFromRow` (row → domain).

### P2-2: `scanEventFromRow` is completely untested

**File:** `src/contexts/guest/infrastructure/mappers/guest.mapper.test.ts`

The test file only covers `scanEventToRow`, `ratingToRow`, and `feedbackToRow`. The `scanEventFromRow` function (the inverse mapping, used when reading from DB) has zero tests. This is the critical path that populates `staffId` from the database row.

### P2-3: `getStaffIdForSession` test asserts `.not.toBeNull()` instead of exact value

**File:** `src/contexts/guest/application/use-cases/get-staff-id-for-session.test.ts:39`

```ts
expect(result).not.toBeNull()
```

The test should assert `expect(result).toEqual(staffId('staff-1'))` — an exact value match. A function returning `staffId('wrong-id')` would pass this test.

### P2-4: Mapper tests don't test `staffId` with non-null values

**File:** `src/contexts/guest/infrastructure/mappers/guest.mapper.test.ts`

All fixture builders default to `staffId: null`. Neither `scanEventToRow` nor `ratingToRow` nor `feedbackToRow` is tested with a non-null `staffId`, despite this being the critical new field.

### P2-5: `recordScanWithRef` first test asserts `.not.toBeNull()` instead of exact staffId

**File:** `src/contexts/guest/application/use-cases/record-scan-with-ref.test.ts:84`

```ts
expect(guestRepo.scans[0].staffId).not.toBeNull()
```

Should assert `expect(guestRepo.scans[0].staffId).toEqual(staffId('user-1'))` to verify the correct staff member is attributed. Currently any non-null staffId passes.

## Low Issues (P3)

### P3-1: `generateReferralCode` test comment contradicts implementation

**File:** `src/contexts/staff/domain/referral-code.test.ts:19-24`

The test says "uses first initial + last name" but the implementation actually uses `first-full + last-full` (e.g., `jane-doe`). The test assertion on line 24 correctly checks for `'jane-doe'`, but the description is misleading. Not a bug, but confusing for maintainers.

### P3-2: `resolveReferralCode` test for "no userId" is misleading

**File:** `src/contexts/staff/application/use-cases/resolve-referral-code.test.ts:60-77`

The test is titled "returns null when assignment has no userId" but the fake repo returns `null` from `findByReferralCode`, making this identical to the "not found" test. It doesn't actually test the case where an assignment exists but has no/empty userId. The `as string` cast on line 14 of the source (`staffId(assignment.userId as string)`) is untested for edge cases.

### P3-3: `buildRating`/`buildFeedback` constructor tests don't test `staffId` propagation

**File:** `src/contexts/guest/domain/constructors.test.ts`

The rating and feedback constructor tests only use `staffId: null` in fixtures and don't assert that a non-null `staffId` is preserved in the built entity. Given `staffId` is a new nullable field, both null and non-null paths should be tested.

### P3-4: `recordScanFn` (server function) has no test

**File:** `src/contexts/guest/server/public.ts`

The server function `recordScanFn` integrates schema validation, header parsing, IP hashing, context resolution, and the `recordScanWithRef` call. While server functions are traditionally hard to unit test, the critical referral code parsing logic (`data.referralCode ?? null`) is only exercised here and has no test coverage. At minimum, the Zod schema should be tested for the referral code field handling (null, undefined, empty string, valid code).

## Positive Findings

- **Consistent use of in-memory fakes** over mocks — all handler tests use state-based assertions on captured readings, which is the right approach per project conventions.
- **Error suppression tests** are present in every metric event handler — good defensive coverage.
- **`generateReferralCode` tests** are thorough for edge cases: empty string, whitespace, single name, special characters, lowercase normalization.
- **`buildStaffAssignment` constructor** correctly tests both with and without optional fields (`portalId`, `referralCode`), verifying defaults to `null`.
- **`getStaffIdForSession`** covers all three relevant states: found with staffId, found without staffId, not found.
- **Event constructors** (`scanRecorded`, `ratingSubmitted`, `feedbackSubmitted`) all test both null and non-null staffId variants.
- **`getPortalQrUrl`** correctly tests URL construction with and without referral codes, including `encodeURIComponent`.
- **`recordScan` silent failure test** is well-structured and verifies no events are emitted on repo error.

## Missing Test Cases

1. **`recordScanWithRef` with empty string referralCode** — verify `""` is treated as no referral code (no repo lookup).
2. **`recordScanWithRef` — exact staffId value assertion** — change `.not.toBeNull()` to exact `staffId('user-1')`.
3. **`submitRating` with staffId** — test that staffId from input propagates to the stored rating and emitted event (requires fixing P0-1 first).
4. **`submitFeedback` with staffId** — same as above for feedback.
5. **`scanEventFromRow` round-trip** — test database row → domain object, including non-null staffId.
6. **Mapper tests with non-null staffId** — `scanEventToRow`, `ratingToRow`, `feedbackToRow` all need a test with `staffId: staffId('staff-1')`.
7. **`onScanRecorded` with non-null staffId** — assert exact staffId value passes through to metric reading.
8. **`onRatingSubmitted` with non-null staffId** — same.
9. **`onFeedbackSubmitted` with non-null staffId** — same.
10. **`getStaffIdForSession` exact value assertion** — assert `staffId('staff-1')` not just `.not.toBeNull()`.
11. **`resolveReferralCode` with empty org ID** — verify behavior with edge-case orgId values.
12. **Integration test: full staff attribution flow** — referral code → resolve → record scan → get staffId → verify metric.
13. **`buildRating`/`buildFeedback` with non-null staffId** — verify propagation through constructors.
14. **`recordScanFn` Zod schema** — test referralCode field with `null`, `undefined`, `""`, valid string.
15. **`submitFeedback` in `public.ts` does NOT call `getStaffIdForSession`** — verify this is intentional or add the call (currently feedback has no staff attribution in the server function either).
