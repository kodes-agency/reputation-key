# Review: Phase 14.5 Schema, Data Integrity & Tenant Isolation

**Reviewer**: Independent code review (schema/data-integrity focus)
**Date**: 2026-05-22
**Phase**: 14.5 — Staff Attribution

## Summary

Phase 14.5 adds `staff_id` (nullable varchar) columns to `scan_events`, `ratings`, `feedback`, and `metric_readings` tables, introduces the `staff_assignments` table with `referral_code`, and wires up referral-code-based attribution through the guest scan → rating → feedback pipeline. The implementation is **solid overall** — branded types, consistent mapper patterns, tenant-isolated queries, and good test coverage. However, there are **two critical issues** (tenant-unsafe `getLatestScanBySession`, referral code uniqueness scoped globally instead of per-org), and several medium/low issues around missing indexes, FK integrity, and collision handling that should be addressed before production.

---

## Critical Issues (P0)

### P0-1: `getLatestScanBySession` queries by `session_id` only — not tenant-safe

**File**: `src/contexts/guest/infrastructure/repositories/guest-interaction.repository.ts` (line 47–56)

```ts
getLatestScanBySession: async (sessionId) => {
  const [row] = await db
    .select()
    .from(scanEvents)
    .where(eq(scanEvents.sessionId, sessionId)) // ← NO organizationId filter
    .orderBy(desc(scanEvents.createdAt))
    .limit(1)
  return row ? scanEventFromRow(row) : null
}
```

**Problem**: This query filters on `session_id` alone. If two organizations generate the same session ID (which is a `varchar` — not a UUID — so collision is plausible via browser fingerprinting bugs or test data), one org's scan could be returned to another org. The returned `ScanEvent` contains `organizationId`, `propertyId`, and `staffId` — leaking cross-tenant data.

**Port definition** (`guest-interaction.repository.ts`, line 18) also takes only `sessionId`:

```ts
getLatestScanBySession(sessionId: string): Promise<ScanEvent | null>
```

**Impact**: The `getStaffIdForSession` use case delegates to this method and returns `staffId` to the caller, which then gets written into the current org's rating/feedback. A colliding session ID could cause a staff attribution from Org A to be written into Org B's records.

**Fix**: Add `organizationId` parameter to the port and filter in the query:

```ts
getLatestScanBySession: async (organizationId, sessionId) => {
  const [row] = await db
    .select()
    .from(scanEvents)
    .where(
      and(
        eq(scanEvents.organizationId, unbrand(organizationId)),
        eq(scanEvents.sessionId, sessionId),
      ),
    )
    .orderBy(desc(scanEvents.createdAt))
    .limit(1)
  return row ? scanEventFromRow(row) : null
}
```

### P0-2: `referral_code` unique constraint is global, not per-organization

**File**: `src/shared/db/schema/staff-assignment.schema.ts` (line 23)

```ts
referralCode: varchar('referral_code', { length: 50 }).unique(),
```

**Problem**: The `.unique()` constraint creates a **global** unique index on `referral_code`. This means two different organizations cannot have staff with the same referral code. While `findByReferralCode` correctly filters by `orgId` via `baseWhere`, the DB constraint prevents legitimate cross-org codes from being created.

With the current 4-char hex hash (65,536 possible values), the probability of collision across tenants grows quickly. This will cause `INSERT` failures when two orgs happen to generate the same `{slug}-{hash}`.

**Fix**: Replace the global `.unique()` with a partial unique index scoped to non-deleted rows within an org:

```ts
// Remove: referralCode: varchar('referral_code', { length: 50 }).unique(),
// Keep nullable column, add composite unique index:
uniqueIndex('staff_assignments_referral_code_unique')
  .on(t.organizationId, t.referralCode)
  .where(sql`referral_code IS NOT NULL AND deleted_at IS NULL`)
```

This also aligns with the pattern already used for `staff_assignments_org_user_property_team_unique`.

## High Issues (P1)

### P1-1: No `staff_id` foreign key — orphaned references on staff assignment deletion

**Files**: `guest.schema.ts`, `metric.schema.ts`

The `staff_id` column in `scan_events`, `ratings`, `feedback`, and `metric_readings` is a plain `varchar` with no foreign key constraint to any staff table. The `staff_assignments` table uses soft-delete (`deletedAt`), so the assignment row persists. However:

- If the `staff_assignments` row is hard-deleted (DBA operation, cascade from another table), the `staff_id` values in guest/metric tables become dangling references.
- The `staff_id` stores `userId` (not `staffAssignmentId`), per `record-scan-with-ref.ts` line 32: `staffId(assignment.userId as string)`. So it references the user, not the assignment. There is no `users` table in this schema for FK enforcement.

**Recommendation**: At minimum, document that `staff_id` references `auth.users.id` or whatever user identity store exists. Consider adding a comment in the schema. If there's a `users` table in another schema, an FK with `ON DELETE SET NULL` would be ideal.

### P1-2: Referral code collision not handled at insert time

**File**: `src/contexts/staff/domain/referral-code.ts`

The `generateReferralCode` function generates a `{slug}-{4-hex-chars}` code. With 4 hex chars = 16 bits = 65,536 possible hash values, and the slug being deterministic per name, two staff members named "Jane Doe" in the same org will generate codes like `jane-doe-a3f2` and `jane-doe-7b1c`. The 4-char hash differentiates them.

However, with 65,536 possible hashes, the birthday problem gives ~50% collision probability at ~300 staff with the same name. There is **no retry logic** — if a collision occurs, the DB unique constraint (currently global, per P0-2) will reject the insert with an opaque Postgres error.

**Fix**: Add retry logic in the service layer:

```ts
async function generateUniqueReferralCode(
  name: string,
  orgId: OrganizationId,
): Promise<string> {
  for (let attempt = 0; attempt < 5; attempt++) {
    const code = generateReferralCode(name)
    const existing = await staffRepo.findByReferralCode(orgId, code)
    if (!existing) return code
  }
  throw staffError('conflict', 'Could not generate unique referral code')
}
```

Alternatively, increase the hash to 6+ chars (16M space) to make collisions astronomically unlikely.

## Medium Issues (P2)

### P2-1: Missing index on `staff_id` columns for query performance

**Files**: `guest.schema.ts`, `metric.schema.ts`

The `staff_id` column is added to `scan_events`, `ratings`, `feedback`, and `metric_readings` but none have indexes. If the application needs to query "all scans for staff member X" or "all metrics attributed to staff Y", these will be full table scans.

**Recommendation**: Add indexes on `staff_id` where analytics queries are expected:

```ts
// In metric.schema.ts — most likely query target
index('metric_readings_staff_idx').on(t.organizationId, t.staffId),
```

For `scan_events` and `ratings`, assess query patterns before indexing.

### P2-2: Missing index on `session_id` in `scan_events`

**File**: `guest.schema.ts`

The `getLatestScanBySession` query filters by `session_id` with `ORDER BY created_at DESC LIMIT 1`. There is no index on `session_id`. As the `scan_events` table grows, this query will degrade.

**Recommendation**:

```ts
index('scan_events_session_idx').on(t.sessionId),
// Or composite:
index('scan_events_org_session_idx').on(t.organizationId, t.sessionId),
```

Note: once P0-1 is fixed to include `organizationId`, the composite index is preferred.

### P2-3: `metricReadings.staffId` inserted as branded type without unbrand

**File**: `src/contexts/metric/infrastructure/repositories/metric.repository.ts` (line 58)

```ts
staffId: reading.staffId,  // StaffId | null — branded type
```

Other repositories consistently use `unbrand()` or `as string` when inserting. The `metric.repository.ts` passes the branded `StaffId` directly. This works at runtime because Drizzle/Pg treats branded strings as strings, but it's inconsistent with the codebase convention and could break if the type system is tightened.

**Compare** with `guest.mapper.ts` line 34: `staffId: scan.staffId as string | null`

**Fix**: Be consistent — either use `reading.staffId as string | null` or `unbrand()`.

### P2-4: `scanEvents` table lacks `organizationId` index

**File**: `guest.schema.ts`

The `scan_events` table has `organizationId` but no index on it. Other tables (e.g., `metricReadings`, `staffAssignments`) have `organizationId` indexes. As data grows, queries filtering by org will be slow.

### P2-5: No `ratingFromRow` / `feedbackFromRow` mappers exist

**File**: `src/contexts/guest/infrastructure/mappers/guest.mapper.ts`

There are `scanEventFromRow`, `scanEventToRow`, `ratingToRow`, and `feedbackToRow` — but no `ratingFromRow` or `feedbackFromRow`. This means ratings and feedback are currently only inserted, never read back via mapper. If they're ever read from DB (e.g., for admin views), these mappers will need to be created. The `staffId` field handling in `scanEventFromRow` is correct (`row.staffId ? staffId(row.staffId) : null`), so the pattern exists.

## Low Issues (P3)

### P3-1: `staffId` type ambiguity — stores `userId`, not `staffAssignmentId`

**File**: `src/contexts/guest/application/use-cases/record-scan-with-ref.ts` (line 32)

```ts
resolvedStaffId = staffId(assignment.userId as string)
```

The `staff_id` column in guest/metric tables stores the `userId` from the staff assignment, not the `staffAssignmentId`. The column is named `staff_id` and the type is `StaffId`, which could mislead developers into thinking it references a staff assignment record rather than a user. This is a semantic naming concern, not a bug.

### P3-2: Mapper tests don't test non-null `staffId` round-trip

**Files**: `guest.mapper.test.ts`, `staff-assignment.mapper.test.ts`

The guest mapper tests only test `staffId: null`. There is no test verifying that a non-null `staffId` survives the `toRow` → `fromRow` round-trip. The staff-assignment mapper tests also don't test `referralCode` round-trip with non-null values or `portalId` round-trip.

### P3-3: `buildSlug` doesn't truncate — very long names produce long codes

**File**: `src/contexts/staff/domain/referral-code.ts`

If a staff member's name is very long, the slug will also be very long. The `referral_code` column is `varchar(50)`, so extremely long names could cause truncation at the DB level. Consider truncating the slug to a reasonable length (e.g., 20 chars) before appending the hash.

### P3-4: `db:push` for dev only — production needs proper migration

The task context mentions `db:push` is used for dev. This is fine for development but **must not** be used in production. Adding nullable columns (`staff_id`) is generally safe for migrations, but:

- The `staff_assignments` table is new and can be created cleanly.
- The `referral_code` unique constraint needs careful handling (see P0-2).
- No data migration is needed since `staff_id` defaults to `NULL`.

**Recommendation**: Generate a proper Drizzle migration (`drizzle-kit generate`) before deploying to staging/production.

### P3-5: `staff_assignments` unique index includes nullable `teamId` and `portalId`

**File**: `staff-assignment.schema.ts` (line 34–38)

```ts
orgUserPropertyTeamUnique: uniqueIndex(...)
  .on(t.organizationId, t.userId, t.propertyId, t.teamId, t.portalId)
  .where(sql`deleted_at IS NULL`)
```

In SQL, `NULL != NULL`, so two rows with `teamId = NULL` can coexist under this unique index. This means the same `(orgId, userId, propertyId)` pair can have multiple direct assignments (no team). This may be intentional (e.g., same user assigned to same property via different portals), but should be documented.

## Positive Findings

1. **Branded ID types**: `StaffId`, `StaffAssignmentId` prevent accidental mixing. All ID types are properly constructed via factory functions.

2. **Tenant isolation in staff repo**: Every query in `staff-assignment.repository.ts` uses `baseWhere(staffAssignments, orgId)` which filters by `organizationId AND deleted_at IS NULL`. `findByReferralCode` correctly includes org filtering.

3. **Tenant guard on insert**: `staffAssignment.insert` checks `assignment.organizationId !== orgId` and throws before DB write. Good defense-in-depth.

4. **Mapper null handling**: `scanEventFromRow` correctly handles nullable `staffId` with ternary (`row.staffId ? staffId(row.staffId) : null`). `staffAssignmentFromRow` uses `!= null` for UUID teamId/portalId (correct — distinguishes null from empty string).

5. **Referral code generation**: Uses `crypto.randomBytes` (cryptographic randomness), not `Math.random`. Handles edge cases (empty string, whitespace-only).

6. **Soft-delete pattern**: `staffAssignments` uses `deletedAt` with `baseWhere` filtering — no data loss on staff removal, and attribution history is preserved.

7. **Test coverage**: Good unit test coverage for mappers, domain logic, and use cases. The metric repo test explicitly verifies tenant isolation.

8. **`recordScanWithRef` correctly scopes referral lookup to `organizationId`**: Line 28 passes `input.organizationId` to `findByReferralCode`, preventing cross-tenant referral code abuse.

9. **Consistent nullable column design**: `staffId` is nullable in all tables — correct, since pre-14.5 data has no attribution and scans without referral codes should have `NULL`.
