# Review: Phase 14.5 Architecture & Domain Integrity

**Reviewer:** Independent Code Review (Hermes Agent)
**Date:** 2026-05-22
**Phase:** 14.5 — Staff Attribution
**Focus:** Architecture, Layer Boundaries, Domain Integrity

## Summary

Phase 14.5 introduces staff attribution via referral codes: scans resolve a referral code to a `StaffId`, ratings/feedback look up the staff from the session's latest scan, and metrics propagate `staffId` throughout. The overall architecture follows hexagonal conventions well — ports in application layer, adapters in infrastructure, domain constructors validate and return `Result` types. However, there are **two critical integration gaps** that render the staff attribution feature non-functional for ratings and feedback: the `getStaffIdForSession` use case is never wired into the build, and both `submitRating`/`submitFeedback` hardcode `staffId: null`. These must be fixed before the phase can be considered complete.

---

## Critical Issues (P0)

### P0-1: `getStaffIdForSession` use case never wired — runtime crash

- **File:** `src/contexts/guest/build.ts`
- **Evidence:** The `useCases` object returned by `buildGuestContext` does not include `getStaffIdForSession`. It wires `recordScan`, `recordScanWithRef`, `submitRating`, `submitFeedback`, `trackReviewLinkClick`, `resolveLinkAndTrack`, `resolvePortalContext`, and `getPublicPortal` — but `getStaffIdForSession` is absent.
- **Impact:** `src/contexts/guest/server/public.ts` line 153 calls `useCases.getStaffIdForSession(sessionId)`. Since `getStaffIdForSession` is never added to the use cases object, this will throw `TypeError: useCases.getStaffIdForSession is not a function` at runtime whenever a guest submits a rating. This is a **blocking runtime error** on the rating submission path.
- **Fix:** Import `getStaffIdForSession` in `build.ts` and wire it:
  ```ts
  import { getStaffIdForSession } from './application/use-cases/get-staff-id-for-session'
  // ...
  getStaffIdForSession: getStaffIdForSession({ guestRepo }),
  ```

### P0-2: `submitRating` and `submitFeedback` hardcode `staffId: null` — staff attribution broken

- **File:** `src/contexts/guest/application/use-cases/submit-rating.ts` (lines 48, 66)
- **File:** `src/contexts/guest/application/use-cases/submit-feedback.ts` (lines 46, 64)
- **Evidence:**
  - `submitRating` builds the rating with `staffId: null` (line 48) and emits the event with `staffId: null` (line 66), regardless of any input. The `SubmitRatingInput` type does not even include a `staffId` field.
  - `submitFeedback` builds feedback with `staffId: null` (line 46) and emits the event with `staffId: null` (line 64). The `SubmitFeedbackInput` type does not include `staffId` either.
  - Both domain constructors (`buildRating`, `buildFeedback`) correctly accept `staffId` in their input types and would propagate it — but the use cases never pass it through.
- **Impact:** Even if P0-1 is fixed and `getStaffIdForSession` is wired, the staff ID is **never propagated** to ratings or feedback. Staff attribution for ratings and feedback is completely non-functional. All ratings and feedback will have `staffId: null` in the database and in emitted events.
- **Fix:** Add `staffId: StaffId | null` to `SubmitRatingInput` and `SubmitFeedbackInput`, pass it through to the domain constructors and event emitters.

### P0-3: `submitRatingFn` server function passes `staffId` to use case that doesn't accept it — type error

- **File:** `src/contexts/guest/server/public.ts` (line 162)
- **Evidence:** Line 162 passes `staffId: staffIdForSession` in the `submitRating` input object, but `SubmitRatingInput` in `submit-rating.ts` does not include a `staffId` field. This would be a TypeScript compilation error (excess property check or unrecognized property depending on strictness).
- **Impact:** Either the code fails to compile (blocking CI), or the field is silently dropped if the type is widenend. Either way, this confirms the feature is incomplete.

## High Issues (P1)

### P1-1: Cross-context dependency — guest imports staff port directly

- **File:** `src/contexts/guest/application/use-cases/record-scan-with-ref.ts` (line 1)
- **Evidence:** `import type { StaffAssignmentRepository } from '#/contexts/staff/application/ports/staff-assignment.repository'`
- **Analysis:** The guest context's `recordScanWithRef` use case directly depends on the staff context's repository port (`StaffAssignmentRepository`). While this is a type-only import of an interface (not a concrete implementation), it represents a bounded context boundary violation. The guest context reaches into the staff context's application layer internals.
- **Architectural concern:** If the staff context changes its repository contract, the guest context breaks. Per hexagonal architecture conventions, cross-context communication should go through:
  1. A **public API** surface (use case or query method exposed by the staff context), or
  2. A **port defined by the consuming context** (guest defines what it needs, staff provides an adapter)
- **Current mitigation:** The composition root (`composition.ts`) passes the same `staffRepo` instance to both contexts, so the wiring is correct. But the dependency direction is inverted — guest should not know about staff's internal port.
- **Recommendation:** Extract a `ResolveStaffReferralCode` port in the guest context's application layer, or use the staff context's public API. The staff context's `resolveReferralCode` use case already exists but is not exposed via public API.

### P1-2: `UserId` rebranded as `StaffId` — semantic type mismatch

- **File:** `src/contexts/guest/application/use-cases/record-scan-with-ref.ts` (line 32)
- **File:** `src/contexts/staff/application/use-cases/resolve-referral-code.ts` (line 14)
- **Evidence:**

  ```ts
  // record-scan-with-ref.ts:32
  resolvedStaffId = staffId(assignment.userId as string)

  // resolve-referral-code.ts:14
  return staffId(assignment.userId as string)
  ```

- **Analysis:** `assignment.userId` is already a branded `UserId`. It is being cast to `string` (stripping the brand) and rebranded as `StaffId`. This conflates two distinct domain concepts. A `StaffId` and a `UserId` are semantically different — `StaffId` represents "the staff member attributed to this guest interaction," which in this implementation happens to be the user ID of the assigned staff member.
- **Risk:** If `StaffId` ever diverges from `UserId` (e.g., staff members get their own identity separate from the user table), every `staffId(userId as string)` call becomes a bug. The `as string` cast also bypasses branded type safety.
- **Recommendation:** Either formally document that `StaffId` is an alias of `UserId` in this bounded context (and provide a single canonical `toStaffId(userId: UserId): StaffId` function in the shared domain), or introduce a lookup that returns the proper `StaffId` type.

### P1-3: `scanEventFromRow` mapper casts `source` without validation

- **File:** `src/contexts/guest/infrastructure/mappers/guest.mapper.ts` (line 19)
- **Evidence:** `source: row.source as ScanEvent['source']`
- **Analysis:** `row.source` is typed as `string` from the Drizzle schema (`varchar('source', { length: 10 })`). The mapper casts it directly to `ScanSource` (`'qr' | 'nfc' | 'direct'`). If the database contains an unexpected value (e.g., migration artifact, manual insert), this produces a domain object with an invalid `source` field — silently violating the type contract.
- **Comparison:** The metric repository (`metric.repository.ts` line 32-34) validates `metricKey` against `VALID_METRIC_KEYS` and throws on invalid values. The guest mapper should follow the same pattern.
- **Recommendation:** Validate `source` against the known `ScanSource` values before casting:
  ```ts
  const VALID_SOURCES = new Set(['qr', 'nfc', 'direct'])
  if (!VALID_SOURCES.has(row.source)) {
    throw new Error(`Invalid scan source in DB row: ${row.source}`)
  }
  ```

## Medium Issues (P2)

### P2-1: `generateReferralCode` performs I/O in domain layer

- **File:** `src/contexts/staff/domain/referral-code.ts` (line 12)
- **Evidence:** `const hash = randomBytes(2).toString('hex')` — `randomBytes` from `crypto` is a Node.js I/O operation.
- **Analysis:** Domain layers should be pure (no I/O, no side effects). While `randomBytes(2)` is synchronous in Node.js, it's still an infrastructure concern. Per the project's conventions ("Pure — ID and time are inputs, no side effects"), the randomness should be injected as a dependency.
- **Severity:** Medium because the function is small, synchronous, and unlikely to cause test issues. But it violates the stated architectural principle.

### P2-2: `onReviewLinkClicked` hardcodes `staffId: null`

- **File:** `src/contexts/metric/infrastructure/event-handlers/on-review-link-clicked.ts` (line 20)
- **Evidence:** `staffId: null` — always null.
- **Analysis:** The `ReviewLinkClicked` event type does not carry a `staffId` field (it only has `linkId`, `organizationId`, `portalId`, `propertyId`, `occurredAt`). This is likely by design since review link clicks don't have staff attribution context. However, it should be explicitly documented as a known limitation. When a guest clicks a review link, the metric reading will never have staff attribution even if the original scan did.

### P2-3: Duplicate metric key validation sets

- **File:** `src/contexts/metric/application/use-cases/record-metric.ts` (lines 10-16)
- **File:** `src/contexts/metric/infrastructure/repositories/metric.repository.ts` (lines 23-29)
- **Evidence:** Both files define `BUILT_IN_METRIC_KEYS` / `VALID_METRIC_KEYS` as identical sets. If a new metric key is added, both must be updated.
- **Recommendation:** Define the canonical set once in `domain/types.ts` alongside the `MetricKey` type, and import it everywhere.

### P2-4: `submitFeedbackFn` in `public.ts` does not propagate `staffId`

- **File:** `src/contexts/guest/server/public.ts` (lines 211-221)
- **Evidence:** Unlike `submitRatingFn` (which at least attempts to call `getStaffIdForSession`), `submitFeedbackFn` does not look up the staff ID at all. It passes no `staffId` to `useCases.submitFeedback`.
- **Impact:** Even after fixing P0-1 and P0-2, feedback will still lack staff attribution unless the server function is also updated to call `getStaffIdForSession`.

## Low Issues (P3)

### P3-1: Branded type `as string` casts in mappers

- **File:** `src/contexts/guest/infrastructure/mappers/guest.mapper.ts` (lines 28-34)
- **File:** `src/contexts/staff/infrastructure/mappers/staff-assignment.mapper.ts` (lines 35-39)
- **Evidence:** `organizationId: scan.organizationId as string`, `staffId: scan.staffId as string | null`, etc.
- **Analysis:** These are at the infrastructure boundary (domain → DB row), where stripping brands is expected. The project has an `unbrand()` utility that should be used instead for clarity, but using `as string` is acceptable in this context. Inconsistent usage (some fields use `unbrand()`, others use `as string`) is a minor style issue.

### P3-2: `portalId` column in staff-assignment schema not referenced in domain types consistently

- **File:** `src/shared/db/schema/staff-assignment.schema.ts` (line 22)
- **Evidence:** The schema has a `portalId` column with FK to portals, and the domain type `StaffAssignment` includes `portalId: PortalId | null`. This is correctly mapped in the mapper. Low concern — just noting the schema evolution.

### P3-3: Test fixture `staffId` fields use raw strings

- **File:** `src/shared/testing/fixtures.ts` (lines 209, 225, 243)
- **Evidence:** `staffId: null` — fixtures default to `null` for `staffId`, which is correct for the default case. However, no fixture builder provides a non-null `staffId` override example, reducing test coverage for the staff attribution path in downstream consumers.

### P3-4: `in-memory-staff-assignment-repo.ts` `softDelete` uses `as StaffAssignment` cast

- **File:** `src/shared/testing/in-memory-staff-assignment-repo.ts` (line 57)
- **Evidence:** `} as StaffAssignment` after spreading and overriding `deletedAt`/`updatedAt`. This is safe because the spread produces a complete object, but the cast could be avoided with proper typing.

## Positive Findings

1. **Clean hexagonal layering:** Domain types are pure data (no methods, no classes). Constructors validate and return `Result<_, GuestError>`. No I/O in domain constructors. Application use cases receive deps as function arguments.

2. **Consistent `StaffId` brand usage:** The `StaffId` branded type is used consistently across all domain types (`ScanEvent`, `Rating`, `Feedback`, `MetricReading`), event types (`ScanRecorded`, `RatingSubmitted`, `FeedbackSubmitted`, `MetricRecorded`), and constructor inputs. Fields are `StaffId | null` everywhere. No raw strings appear where `StaffId` is expected.

3. **Event design follows conventions:** Events use `_tag` discriminators, carry all needed data (including nullable `staffId`), and follow `{context}.{action}` naming. The master `DomainEvent` union in `shared/events/events.ts` is kept in sync.

4. **Port/adapter separation is clean:** Repository ports are interfaces (type aliases) in the application layer. Infrastructure implementations use Drizzle mappers. Composition root wires everything. No infrastructure imports in domain or application layers.

5. **Metric context correctly propagates `staffId`:** All metric event handlers (`onScanRecorded`, `onRatingSubmitted`, `onFeedbackSubmitted`) correctly forward `event.staffId` to the `recordMetric` use case. The metric repository correctly maps `staffId` to/from DB rows.

6. **Referral code design is sound:** The `generateReferralCode` function produces deterministic slugs with random hashes. The `findByReferralCode` repository method is properly tenant-isolated via `baseWhere`.

7. **Exhaustive pattern matching:** `public.ts` uses `ts-pattern`'s `.exhaustive()` for error code → HTTP status mapping. The master `DomainEvent` union ensures all event types are handled.

8. **Test coverage is thorough:** All new use cases, constructors, mappers, and event handlers have corresponding test files with meaningful assertions.
